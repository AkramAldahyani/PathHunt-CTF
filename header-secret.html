HTTP Headers

Category: Protocol · Technique: Header inspection
Brief

Every HTTP response carries headers — invisible key/value pairs that travel alongside the HTML. Servers use them to send instructions, version info, and sometimes things they didn't mean to expose.

This page's response carries something extra alongside the HTML that the renderer never paints. It's not in the DOM. It's not in any file you can right-click. But every request you've ever made has come bundled with this kind of metadata.

Hint The HTML body is only half of an HTTP response. Look at what wraps it — the part the browser parses before it ever reaches the page. Any tool that speaks HTTP can show it to you, and your browser ships with one. Custom additions usually wear an X- prefix.

If you opened the page from the filesystem, you'll see no response at all — there is no server. Serve the project over HTTP first.

Sample response (a different page)

This is what a typical response envelope looks like — not for this page, just an example so you know the shape of what you're hunting for.

Status: 200 OK
Content-Type: text/html; charset=utf-8
Server: pathhunt-edge/1.4
Cache-Control: no-store
X-Powered-By: curiosity
Why it matters

Real-world sites routinely leak information through headers: framework versions (X-Powered-By: Express), internal hostnames, debug tokens, even stack traces on error pages. Reading headers is always step one of recon.

Reading a response envelope is the most basic move in web recon — and the easiest to forget once you fall in love with the rendered page. Anything the server attaches to its response is fair game; the browser will happily show you every byte if you ask it to.
